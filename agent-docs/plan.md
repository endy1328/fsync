# 계획

## 목표

`fsync` 1.0 단방향 동기화 기능을 유지하면서, 설정에 따라 단방향과 양방향을 선택할 수 있는 1.1 확장 설계를 수립하고 구현 준비를 마칩니다.

## 구현 범위

- 기존 1.0 단방향 job 동작 유지
- `mode` 또는 동등한 설정 기반의 job 타입 분기 지원
- 2개 디렉토리 간 양방향 sync를 위한 설정, 상태 저장, 계획 수립, 실행 구조 설계
- 충돌 감지, 삭제 전파, 초기 sync 정책 정의
- 1.1 구현을 위한 단계별 작업 순서와 완료 조건 정리
- 양방향 핵심 시나리오 검증 계획 수립

## 구현 결정 사항

- 설정 형식은 TOML로 유지하고 `tomllib`를 사용합니다
- 변경 감지는 polling 방식만 사용합니다
- 1.0 단방향 엔진은 유지하고, 양방향 엔진은 별도 계층으로 추가합니다
- 동시성은 계속 job 단위로 처리합니다
- 양방향은 2-peer 모델만 지원하고 mesh sync는 제외합니다
- 양방향 삭제 전파는 상태 저장소를 전제로 합니다
- 양방향 충돌은 자동 머지하지 않고 명시적 충돌 정책으로 처리합니다
- busy 파일은 파일 단위로 보류하고 다음 주기에 재시도합니다
- target 또는 peer 쓰기 실패는 현재 job을 즉시 실패시킵니다
- 현재 구현은 기존 초안을 폐기하지 않고 확장하는 방식입니다
- `mode`가 없으면 기존 호환을 위해 `one_way`로 간주합니다
- 양방향 state는 1.1에서 JSON 파일로 저장하고 atomic rename으로 갱신합니다
- `AppConfig.jobs`는 one-way와 bidirectional job union을 허용하는 방향으로 확장합니다

## 1.1 설계 쟁점

- 설정 스키마를 기존 `source`/`targets` 중심 모델과 어떻게 공존시킬지 결정합니다
- 양방향 job의 기준 상태 저장 포맷을 JSON으로 시작하고, 필요 시 후속 버전에서 sqlite 확장을 검토합니다
- 초기 sync 기본값은 `manual`로 고정합니다
- 삭제 전파 기본값은 `tracked`로 고정합니다
- 충돌 기본값은 `keep_both`로 고정합니다

## 작업 단계

1. 1.1 목표와 비목표를 확정합니다.
2. 양방향 설정 스키마와 상태 저장 구조를 설계합니다.
3. 상태 저장은 JSON, 초기 sync는 `manual`, 충돌 정책은 `keep_both`, 삭제 정책은 `tracked`로 확정합니다.
4. `OneWayJobConfig`, `BidirectionalJobConfig`, `BidirectionalState`, `BidirectionalPlan` 기준 타입을 확정합니다.
5. 모듈 구조와 책임 분리를 확정합니다.
6. 구현 단계를 config, state/planner, executor, tests, docs 순으로 분해합니다.
7. 단계별 완료 조건과 회귀 검증 조건을 정의합니다.

## Agent 배분

- 메인 에이전트: 범위 확정, 설계 선택, 최종 통합, 최종 보고
- 계획 에이전트: 1.1 단계 분해, 의존성 정리, 완료 조건 정리
- 판단 에이전트: 충돌 정책, 삭제 정책, 초기 sync 정책 검토
- 실행 에이전트 A: config schema, CLI 계약, 예제 설정, README 변경
- 실행 에이전트 B: state 저장소, planner, bidirectional engine 구현
- 실행 에이전트 C: 테스트 추가, 회귀 확인, 실패 시나리오 검증
- 검증 에이전트: 데이터 유실 위험, state 커밋 시점, 정책 누락, 테스트 공백 검토

## 검증 계획

- 단방향 1.0 회귀 테스트가 계속 통과하는지 확인합니다
- 양방향 설정 로드가 정상 동작하는지 확인합니다
- 좌측 신규 파일이 우측으로, 우측 신규 파일이 좌측으로 전파되는지 확인합니다
- 한쪽만 수정된 파일이 반대편으로 전파되는지 확인합니다
- 양쪽 동시 수정이 충돌로 감지되는지 확인합니다
- 상태 저장 후 변경 없는 재실행에서 noop이 되는지 확인합니다
- 상태 기반 삭제 전파가 의도한 경우에만 일어나는지 확인합니다
- 삭제와 수정이 동시에 발생할 때 충돌로 처리되는지 확인합니다
- busy 파일이 해당 파일만 보류하고 다른 파일은 진행되는지 확인합니다
- 실패 시 state가 잘못 커밋되지 않는지 확인합니다

## 남은 작업

- 설정 스키마를 실제 `config.py` 타입으로 옮깁니다
- JSON state 파일 직렬화/역직렬화 구조를 구현합니다
- planner action/plan 타입을 코드로 옮깁니다
- bidirectional executor의 실패 시 state 미커밋 경계를 구현합니다
- 설계 확정 후 구현 작업을 단계별로 시작합니다
