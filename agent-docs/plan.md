# 계획

## 목표

기존 초안 코드를 기반으로 현재 `fsync` v1 구현을 완성하고, Python 3.12.3 기반 단방향 파일 동기화 애플리케이션으로 유지합니다.

## 구현 범위

- `[[jobs]]` 구조의 TOML 설정
- `source` 1개에서 하나 이상의 `targets`로 가는 단방향 sync
- `run`, `once` 명령을 사용하는 polling 기반 실행
- job 단위 병렬 실행
- 각 target의 실제 파일 상태 비교를 통한 변경 파일만 복사
- `copy_deleted`를 통한 선택적 삭제 반영
- busy source 파일 보류 후 재시도
- target 실패 시 job 실패 처리
- 핵심 시나리오에 대한 자동 단위 테스트

## 구현 결정 사항

- 설정 형식은 TOML로 유지하고 `tomllib`를 사용합니다
- 변경 감지는 polling 방식만 사용합니다
- 동시성은 target 단위가 아니라 job 단위로 처리합니다
- `copy_deleted`는 job 단위 옵션이며 기본값은 `false`입니다
- busy source 파일은 파일 단위로 보류하고 다음 주기에 재시도합니다
- 메모리 snapshot 대신 각 target의 실제 파일 상태를 기준으로 복사 여부를 판정합니다
- 여러 target 중 일부만 오래되었거나 누락된 경우 그 target에만 복사합니다
- target 쓰기나 삭제 실패는 현재 job을 즉시 실패시킵니다
- 현재 구현은 기존 초안을 폐기하지 않고 확장하는 방식입니다

## 검증 계획

- TOML `[[jobs]]` 설정 로드가 정상 동작하는지 확인합니다
- 신규 파일과 수정 파일이 모든 target으로 복사되는지 확인합니다
- `copy_deleted=true`일 때 target에서도 삭제되는지 확인합니다
- busy source 파일이 보류되고 다음 주기에 재시도되는지 확인합니다
- 재시작 후에도 target과 동일한 파일은 다시 복사되지 않는지 확인합니다
- 여러 target 중 일부 target만 누락된 경우 그 target만 복사되는지 확인합니다
- target 실패가 현재 job을 중단시키고 snapshot이 잘못 전진하지 않는지 확인합니다
- target metadata 실패가 target 측 실패로 처리되는지 확인합니다
- `run_once`가 모든 job 종료를 기다린 뒤 실패를 보고하는지 확인합니다

## 남은 작업

- 필요한 경우 설정 검증과 경계 케이스 테스트를 더 추가합니다
- 단위 테스트 외에 실제 config를 이용한 CLI 수동 점검을 수행합니다
- 운영 요구가 생기면 관측성 기능을 확장합니다
- 대상 OS에서 실제 파일 잠금 동작을 수동으로 점검합니다
