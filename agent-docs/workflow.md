# 작업 흐름

## 목적

- 이 문서는 이 저장소에서 메인 에이전트와 서브에이전트가 어떻게 협업하는지 정의합니다.
- 사용자가 명시적으로 다중 에이전트 조율을 요청한 경우 계획, 판단, 실행, 검증, 상태 에이전트를 병렬로 운용합니다.
- 계획, 실행, 검증, 상태 보고를 분리해서 유지합니다.
- 다음 작업자가 별도 설명 없이도 흐름을 이해할 수 있게 합니다.

## 기본 흐름

1. 메인 에이전트가 현재 저장소 상태와 사용자 제약을 확인합니다.
2. 메인 에이전트가 남은 작업을 정리합니다.
3. 실행 순서가 불명확하면 계획 에이전트를 생성합니다.
4. 설계나 범위에 영향을 주는 선택이라면 판단 관점 3개를 수집합니다.
5. 안전하게 분리 가능한 작업은 소유 범위를 나눠 실행 서브에이전트를 만듭니다.
6. 메인 에이전트가 결과를 통합하고 저장소 변경을 마무리합니다.
7. 검증 에이전트가 완료된 작업을 검토합니다.
8. 상태 에이전트가 현재 체크포인트를 갱신합니다.
9. 메인 에이전트가 최종 결과를 정리해 보고합니다.

## 작업 접수

- 목표, 제약, 명시적 제외 범위를 먼저 분리합니다.
- 현재 작업이 문서 작업인지 구현 작업인지 확인합니다.
- 각 턴의 목표는 구체적인 산출물 하나로 제한합니다.

## 계획

- 작업을 작은 단계로 나누고 각 단계의 결과물을 명확히 합니다.
- 의존성과 완료 조건을 기록합니다.
- 프로젝트 단계가 요구하면 기능 작업보다 운영 문서 정리를 우선합니다.

## 판단

- 범위, 구조, 절차에 실질적인 영향을 주는 판단에는 3개의 독립 관점을 사용합니다.
- 각 관점의 공통점과 차이를 비교합니다.
- 최종 결정은 메인 에이전트가 내리고 문서에 반영합니다.

## 실행

- 각 실행 에이전트는 명확히 제한된 파일이나 책임 범위를 가집니다.
- 실행 에이전트는 임의로 범위를 확장하지 않습니다.
- 범위가 바뀌면 먼저 계획 또는 판단 단계로 되돌아갑니다.

## 검증

- 검증은 마지막 한 번이 아니라 의미 있는 작업 단위마다 수행합니다.
- 범위 적합성, 문서 충돌, 누락된 갱신, 명확한 위험을 검토합니다.
- 수정이 들어간 경우 다시 검증합니다.

## 상태 보고

- 문서 범위, 현재 단계, 진행 중 작업, 다음 작업, blocker를 추적합니다.
- 상태 보고는 짧고 사실 중심으로 유지합니다.
- 상태 문서는 계획 문서나 검증 기록의 대체물이 아닙니다.

## 문서 유지 규칙

- 사용자 관점의 시작점이 바뀌면 `README.md`를 갱신합니다.
- 운영 규칙이 바뀌면 `agent.md`를 갱신합니다.
- 실행 순서가 바뀌면 `agent-docs/plan.md`를 갱신합니다.
- 현재 상태가 바뀌면 `agent-docs/status.md`를 갱신합니다.
- 검토 기준이 바뀌면 `agent-docs/verification.md`를 갱신합니다.
- 검증 실행 결과나 self-test 산출물은 `reports/`에 둡니다.
- 이정표나 중요한 결정이 생기면 `agent-docs/worklog.md`를 갱신합니다.
